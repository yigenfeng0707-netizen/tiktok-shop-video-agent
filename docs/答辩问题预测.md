## 一、场景价值类（25 分）

### Q1: 这个痛点真的是跨境卖家最痛的吗？有没有调研数据支撑？

**参考回答**：根据场景说明文档，跨境卖家四大痛点中"视频生产成本高"和"多国本地化周期长"是公认的核心痛点。单条视频拍摄成本 300-800 元、月均支出 1-4 万元，5 国全量上线需 2-3 周，这些数据来自行业普遍认知。同时长尾 SKU 无素材导致 GMV 损失 40%+，海量 SKU（100-1000+）与实际产能不足需求 10% 的剪刀差是卖家最直接的经营损失。痛点排序的依据是"成本可量化 + 转化率直接受损"双维度，而非主观判断。

### Q2: 为什么聚焦 TikTok Shop 而不是亚马逊或其他平台？

**参考回答**：TikTok Shop 是内容电商，视频是核心流量入口，与亚马逊"搜索 + 详情页"的逻辑完全不同。TikTok Shop 跨境 GMV 2024 年 200-300 亿美元、年增长率 100%+，且覆盖 5 个高增长国家（美/越/沙/英/印尼），多国本地化需求天然强烈。亚马逊主图视频偏静态展示，对多国本地化、文化禁忌规避的需求弱。本项目选题紧贴"内容电商 + 多国跨境"的双重红利，场景适配度最高。

### Q3: 目标用户月 GMV 5-50 万，这个范围是不是太宽了？

**参考回答**：5-50 万 GMV 区间看似较宽，但其共同特征是"团队 1-5 人、视频制作能力薄弱、依赖外包或自拍"，痛点高度一致。5 万以下卖家处于冷启动期，预算有限更需要 AI 降本；50 万以上卖家有稳定出单，更需要批量产能覆盖长尾 SKU。两个层级对"成本降低 80%、产能提升 10 倍"的价值诉求一致，所以放在一起是合理的。后续 SaaS 套餐已通过 299/999/3999 三档分层覆盖，分层正是为了适配区间内不同规模。

### Q4: 50-80 万卖家的数据来源？

**参考回答**：该数据是基于公开行业报告与第三方研究估算的区间值，并非精确统计。参考依据包括 TikTok 官方披露的 Shop 卖家增长趋势、跨境 ERP 服务商（店小秘、马帮等）的客户体量、以及 2024 年 TikTok Shop 跨境 GMV 200-300 亿美元规模反推。我明确这是一个估算区间而非精确数字，文档中也使用"估算 50-80 万家"的表述。即使是下限 50 万家，可服务市场规模（SAM）仍达 75-200 亿元/年，市场容量足以支撑产品商业化。

### Q5: 本地生活经营者为什么也是目标用户？

**参考回答**：TikTok Shop 在东南亚和中东市场与本地生活服务高度融合，餐饮、零售、服务类跨境商家同样面临多国文化定制与多语种内容生产的痛点。但本地生活是次要目标用户，主战场仍是商品带货卖家。文档中将其列入是为了说明场景的延展性，后续可拓展至"本地好物内容"赛道。核心功能（多语种 TTS、文化禁忌规则）对本地生活经营者同样适用，无需额外技术投入。

## 二、商业化潜力类（25 分）

### Q1: 999 元/月的定价依据是什么？卖家愿意付费吗？

**参考回答**：999 元/月对应专业版 200 条/月、5 国本地化。按单条成本 60-150 元计算，200 条价值 1.2-3 万元，SaaS 费用仅占传统成本的 3-8%，ROI 极高。对比传统月均视频支出 1-4 万元，节省 90%+。定价参考了 HeyGen、剪映企业版等同类工具的市场价格区间，并低于其叠加成本。卖家付费意愿有支撑：跨境服务商分销渠道反馈中小卖家对"百元级视频工具"接受度高。入门版 299 元降低试用门槛，企业版 3999 元覆盖规模客户。

### Q2: 按条 9.9 元和 SaaS 999 元，哪个是主力模式？

**参考回答**：SaaS 订阅是主力模式，按条计费是补充。SaaS 模式有稳定 ARR、用户粘性高、LTV 可预测，符合 B 端工具的主流变现路径。按条 9.9 元主要面向试用用户、低频需求个体卖家，以及作为分销渠道的引流产品。预计 SaaS 占比 70-80%，按条计费 20-30%。两种模式共享同一套底层流水线，无额外开发成本，按条计费更像是 SaaS 的"试用转付费"漏斗工具。

### Q3: 跨境服务商分销怎么落地？分润 20% 合理吗？

**参考回答**：分销落地路径是：与店小秘、马帮、领星等跨境 ERP/代运营服务商合作，将本产品作为增值模块嵌入其工作流，卖家通过 ERP 一键调用。分润 20-30% 的依据是行业 SaaS 分销普遍在 20-40% 区间（参考有赞、微盟渠道政策）。20% 是入门分润，规模客户可上浮至 30%。落地步骤：先与 1-2 家 ERP 完成 API 对接 PoC → 试点 50 家卖家 → 3 个月内规模化。MCP 协议已经把能力标准化暴露，对接成本低。

### Q4: 首年 50-100 万收入的测算依据？

**参考回答**：测算基于保守模型：种子期（1-3 月）100 用户、月收入 10 万；成长期（4-6 月）500 用户、月收入 50 万；规模期（7-12 月）2000 用户、月收入 200 万。首年累计 50-100 万取的是种子 + 成长期的保守下限。按 ARPU 1000 元/月估算，100 个付费用户即可达 10 万月收入。获客来源：跨境服务商分销引流、TikTok 卖家社群、行业展会。该测算未计入按条计费和企业定制收入，实际空间更大。

### Q5: 企业定制 3-10 万的项目包含什么？

**参考回答**：企业定制面向月 GMV 50 万+ 的规模卖家和品牌方，包含：①专属数字人形象定制（基于品牌 IP 训练数字人）；②私有化文化禁忌规则库（按品牌品类扩展）；③定制视频模板（BGM、字幕样式、转场风格）；④数据复盘仪表盘对接其 BI 系统；⑤专属国家扩展（如日本、韩国）；⑥SLA 保障与专属客服。3 万为基础包，10 万为全功能包。成本侧主要是数字人训练费 + 定制开发人月，毛利 50%+。

## 三、创新性类（15 分）

### Q1: 多国批量本地化和市面上翻译工具有什么区别？

**参考回答**：本质区别有三点：①翻译工具只翻译文字，本方案同时处理脚本结构、文化元素、视觉呈现（数字人面孔、BGM、字幕风格）三个维度；②翻译工具无文化禁忌规避能力，本方案内置 5 国 15 条文化禁忌规则引擎（如中东女性用品不出现真人模特、印尼食品优先 halal 认证）；③翻译工具无热点融入能力，本方案 Prompt 模板自动融入当地节日、挑战赛、流行梗。一次输入产出 5 国成片 mp4，是端到端生产而非文字翻译。

### Q2: 文化禁忌规则引擎是怎么实现的？覆盖多少场景？

**参考回答**：当前规则引擎基于 `config/cultural_taboos.json` 配置实现，覆盖 3 大区域（中东、欧美、东南亚）5 个国家，共 15 条规则。按维度划分：宗教敏感（伊斯兰/佛教/十字架）、女性着装、女性用品展示、食品 halal 认证、保健品免责声明、夸张用语规避、环保卖点突出、斋月特别规则、左手礼仪等 9 类场景。规则结构包含 id/region/countries/category/rule/action 六个字段，action 触发对应处理逻辑（hide_model / focus_product / add_disclaimer 等）。当前是规则驱动，后续可迭代为 LLM + 规则双引擎。

### Q3: 数据闭环迭代具体怎么反哺？周期多长？

**参考回答**：反哺路径是：发布视频 → 数据复盘子 Agent 采集播放量/完播率/转化率 → GPT-4 分析识别高效脚本模式（hook 类型、BGM 类型、节奏结构）→ 优化建议回写脚本生成 Prompt 模板 → 下次生成更优脚本。反哺周期取决于数据采集频率：当前 Demo 阶段使用 mock 数据演示逻辑；生产环境下，TikTok Shop Seller Center API 数据可按日级采集，反哺周期为 T+1（次日生效）。中长期可建立 A/B 测试机制，多版本脚本并行投放，数据驱动 Prompt 持续迭代。

### Q4: HeyGen 等工具已经有数字人，你的创新在哪？

**参考回答**：HeyGen 是单点工具（数字人合成），本方案是端到端流水线（Agent 编排）。差异点：①HeyGen 需要人工写脚本 + 手动调用 + 手动剪辑，本方案一次输入自动跑完 4 步；②HeyGen 无文化禁忌规则引擎，卖家不懂当地文化容易踩坑，本方案自动规避；③HeyGen 无数据反哺闭环，本方案发布数据回写优化脚本；④HeyGen 单条计费成本高，本方案通过批量编排将单条成本压到 60-150 元。本质是"工具 → Agent 流水线"的升级，而非替代 HeyGen。

### Q5: 和 Dify/Coze 上的工作流有什么本质区别？

**参考回答**：三点本质区别：①定位不同——Dify/Coze 是通用 Agent 平台，需用户自己搭工作流；本方案是垂直场景产品，开箱即用，卖家无需懂 Agent；②领域知识沉淀——本方案内置 5 国文化禁忌规则、本地化 Prompt 模板、视频模板配置、BGM 库，这些是跨境电商领域知识，Dify/Coze 上需从零搭建；③MCP 协议标准化暴露——通过 open.hirebox.cn 对外提供 4 个标准化接口，便于 ERP/服务商集成，Dify 工作流是封闭生态。本方案是"垂直产品"，Dify 是"通用平台"，定位不同。

## 四、体验完成度类（15 分）

### Q1: Demo 是 mock 数据，真实环境下能跑通吗？

**参考回答**：坦诚说明，当前 Demo 阶段数字人合成和数据复盘使用 mock 数据，原因是：①HeyGen API 调用有真实成本，黑客松期间不适合高频调用；②TikTok Shop Seller Center 数据接口需要卖家授权，无法在 Demo 环境获取。但流水线骨架（主编排 + 4 子 Agent + MCP 接口）是真实可执行的，脚本生成、批量剪辑（ffmpeg）两步是真实产出。生产环境只需替换 mock 为真实 API 调用，架构无改动。已预留 `.env.example` 配置真实 API Key 即可切换。

### Q2: 数字人视频实际产出质量如何？

**参考回答**：Demo 阶段未实际调用 HeyGen 产出成片，质量评估基于 HeyGen 官方公开 Demo 和行业评测：多语种 TTS 发音自然度达真人 85%+，口型同步准确率 90%+，面部表情较自然。本方案对数字人质量的增量在于脚本侧——通过本地化 Prompt 和文化禁忌规则，保证脚本内容质量，再交由 HeyGen 合成。真实产出质量瓶颈在 HeyGen 而非本方案。后续上线前会做 50 条样本质量抽检，达标后开放给卖家。

### Q3: ffmpeg 命令是生成的还是真实执行的？

**参考回答**：ffmpeg 命令是真实执行的，不是 mock。批量剪辑子 Agent（`agents/batch_editor.py`）真实调用 ffmpeg 完成：视频拼接、字幕硬编码（SRT 文件已产出在 `assets/subtitles/`）、BGM 混音、分辨率统一（1080×1920）。`assets/subtitles/` 下已有 5 国 × 3 版本共 15 个 SRT 字幕文件作为真实产出证据。ffmpeg 6.0+ 是稳定开源工具，本方案基于其标准命令行接口封装，无 mock 成分。

### Q4: 0.589 秒响应是 mock 的速度，真实 API 调用要多久？

**参考回答**：坦诚说明，0.589 秒是 mock 数据下的端到端响应，真实环境下耗时不同：①脚本生成（GPT-4 调用）单条约 3-8 秒，5 国并发约 8-15 秒；②数字人合成（HeyGen API）单条 30-90 秒，5 国并发约 60-120 秒；③批量剪辑（ffmpeg）单条 10-30 秒，5 国并发约 30-60 秒。真实端到端约 2-5 分钟，仍远快于传统 2-3 周。Demo 的 0.589 秒是为了展示流程完整性，真实性能会在上线前做压测和优化（如异步队列、缓存机制）。

### Q5: 5 国视频实际产出效果如何？

**参考回答**：Demo 阶段已真实产出：5 国 × 3 版本 = 15 个 SRT 字幕文件（位于 `assets/subtitles/`，文件名含国家代码 US/VN/SA/GB/ID 和时间戳），证明脚本生成和字幕环节真实跑通。数字人成片因 API 成本原因使用 mock 路径，但合成逻辑（多语种 TTS、面孔匹配）已实现并通过单元测试。实际效果评估计划：上线前用 5 个真实产品做端到端验证，每国产出 3 条样本，共 15 条，由当地母语用户做内容质量评测（脚本贴合度、文化适配度、口播自然度）。

## 五、技术类问题（10 分）

### Q1: MCP 协议具体怎么工作的？

**参考回答**：MCP（Model Context Protocol）是 Anthropic 提出的 Agent 能力标准化协议。本方案通过 open.hirebox.cn 将 4 个子 Agent 能力封装为 MCP 接口：generate_script（脚本生成）、synthesize_video（数字人合成）、batch_edit（批量剪辑）、review_data（数据复盘）。工作流程：外部调用方（如 MiXer AI 或第三方 ERP）通过 MCP 协议发起请求 → open.hirebox.cn 路由到对应子 Agent → 子 Agent 执行后返回结构化 JSON。优势是接口标准化、解耦实现、支持第三方集成。MCP 链接已上线：https://open.hirebox.cn/mcp/65_my-agent_4Dsd7ZgAud4

### Q2: 为什么用 MiXer AI 而不是自建服务？

**参考回答**：三点理由：①开发效率——黑客松 5 天内 solo 完成 4 子 Agent 编排，自建调度框架不现实，MiXer AI 提供可视化工作流、状态管理、错误重试等开箱即用能力；②MCP 标准化——MiXer AI 原生支持 MCP 协议暴露，免去自建 API 网关；③聚焦业务——将精力集中在 Prompt 模板、文化禁忌规则、剪辑脚本等业务逻辑，而非基础设施。后续规模化后可评估自建调度服务，但当前阶段 MiXer AI 是最优解。技术选型匹配 solo 团队的资源约束。

### Q3: 4 个子 Agent 之间的数据怎么流转？

**参考回答**：数据流转通过主编排 Agent（`orchestrator/main.py`）统一调度，采用 JSON 结构化传递：①步骤 1 输出 scripts 列表（每国一条脚本 JSON，含 hook/body/cta/country 字段）→ ②步骤 2 接收 scripts + countries，过滤掉 error 项后调用数字人合成，输出 videos 列表（含 video_url/country/duration）→ ③步骤 3 接收 videos + product_info，组装为 items 列表调用 ffmpeg 批量剪辑，输出 edited_videos → ④步骤 4 接收 mock_videos 做数据复盘。每步记录耗时和错误，单步失败不中断流水线（错误隔离）。状态通过 `_status` 字典维护（idle/running/done）。

### Q4: 如何保证文化禁忌规则的准确性？

**参考回答**：当前准确性保障三层机制：①规则来源——基于跨境电商行业公开资料、TikTok Shop 官方政策、当地文化常识整理，非主观臆测；②Claude 校验——脚本生成时调用 Claude 做二次安全合规判断，长上下文理解能力强；③规则可配置——所有规则存于 `cultural_taboos.json`，可随时增补修订，无需改代码。当前覆盖 15 条规则是 MVP 阶段的最小可用集，后续计划：邀请当地母语用户做规则评审、对接 TikTok Shop 官方违规案例库、建立规则误判反馈通道，持续扩充规则库至 50+ 条。

### Q5: 数据复盘用真实 TikTok API 还是模拟数据？

**参考回答**：坦诚说明，Demo 阶段使用 `config/mock_data.json` 中的模拟数据，包含 5 国 15 条视频样本（含 video_id/product_name/country/script_summary/hook_type/bgm_type/publish_time 等字段）。复盘逻辑（高效脚本模式识别、优化建议生成）是真实实现的，只是数据源是 mock。生产环境计划对接 TikTok Shop Seller Center API 采集真实数据（播放量、完播率、转化率），需要卖家授权 OAuth。当前已预留数据接入接口，切换真实 API 无需改动复盘逻辑代码。

## 六、路演与协作类（10 分）

### Q1: 个人 solo 参赛，后续怎么扩大团队？

**参考回答**：solo 参赛是验证产品可行性的最快路径，后续扩团队计划分三阶段：①种子期（1-3 月）保持精简，1 人 + AI 工具（Cursor/Claude Code）完成迭代，验证 PMF；②成长期（4-6 月）补充 2 人：1 名视频工程师（ffmpeg 优化、数字人调优）+ 1 名跨境运营（规则库扩充、客户成功）；③规模期（7-12 月）扩至 5-8 人，增加销售、BD、后端工程师。已明确愿意入驻、认证、共创，希望通过赛事对接人才和资源。solo 模式决策快、迭代快，是早期优势而非劣势。

### Q2: 5 分钟讲不完所有内容，怎么取舍？

## 答辩注意事项

### 一、回答原则

1. **诚实优先**：对 Demo 阶段的 mock 数据、API 调用未真实执行等局限要主动坦诚说明，不回避、不粉饰。评委更欣赏诚实 + 改进规划，而非夸大其词。
2. **简洁有力**：每个回答控制在 100-200 字，先给结论，再给 1-2 个支撑数据或案例，避免长篇大论。
3. **数据支撑**：所有论断尽量带数据（成本 300-800 元、产能 30-500 条、卖家 50-80 万家、规则 15 条、5 国 15 字幕文件等），避免"很多""很大"等模糊表述。
4. **结构化表达**：复杂问题用"第一/第二/第三"或"①②③"分点陈述，便于评委抓重点。

### 二、避坑指南

1. **不要过度承诺**：不说"100% 准确""完全替代真人""保证转化率提升 X 倍"等绝对化表述，改用"目标""预计""基于行业基准"等留有余地的表述。
2. **不要贬低竞品**：提到 HeyGen/Dify/Coze 时肯定其能力，强调本方案的差异化定位（垂直场景 + 端到端 + 文化规则），而非"他们不行"。
3. **不要回避短板**：solo 团队、mock 数据、未真实接入 API 等是客观事实，主动说明 + 给出改进路径，比被动暴露更得分。
4. **不要编造数据**：50-80 万卖家是估算区间，明确说明数据来源（行业报告 + 反推），不要包装成精确统计。
5. **不要陷入技术细节**：评委关注场景价值和商业化，技术细节点到为止，除非评委追问。

### 三、高分技巧

1. **用数据说话**：每个回答尽量带 1 个具体数字（成本、产能、规则数、文件数、用户数），数据密度高 = 专业度高。
2. **展示迭代思考**：对每个短板，不仅说明现状，还要给出"当前 → 下一步 → 中长期"的迭代路径，体现产品演进思维。例如文化禁忌规则"当前 15 条 → 30 条 → 50+ 条"。
3. **体现商业化思维**：技术问题回答时也要带商业化视角，如 MCP 协议不仅是技术标准，也是"便于 ERP 集成、降低 BD 成本"的商业基建。
4. **突出 solo 优势**：把"个人参赛"从劣势转化为优势——决策快、24 小时 MVP、AI 工具放大单兵产能，体现新时代的"AI Native 创业"范式。
5. **展示真实产出**：现场指明 `assets/subtitles/` 下 15 个真实 SRT 字幕文件作为产出证据，比口头描述更有说服力。
6. **预留合作钩子**：回答中多次提到"愿意入驻、认证、共创""对接 ERP""寻求资源支持"，为后续合作埋伏笔。
